## 🎮프로젝트 개요
### - 개발 기간
25.01.10 ~ 25.01.16
### - 스파르타 용사로 환생했으니 공주를 구해조!! 아니아니 절대 무리인게 당연하잖아!?란?
이세계로 전송되어 다짜고짜 공주를 구해달라는 부탁을 받은 주인공!

그렇게 모험을 떠나게 되는데... 과연 공주를 구하고 이세계에서 행복하게 살 수 있을까?

### - 게임 설명
캐릭터가 생성되면 몬스터와 싸워 경험치, 골드를 얻으며 점점 더 강해집니다.

점점 강해지는 몬스터들로부터 살아남아 최종 보스와 싸워 이기면 클리어하는 텍스트 RPG 게임입니다!

게임 화면은 전투 시작, 전투 끝, 상점으로 나뉩니다.


**전투 시작**
- 전투가 시작되면 플레이어가 먼저 몬스터를 공격! -> 몬스터가 플레이어를 공격! 하며 자동으로 진행됩니다.

**전투 끝**
- 죽으면 Game Over, 살아남으면 골드와 경험치를 얻고 계속 진행합니다.
- 전투가 끝나면 현재 캐릭터의 상태를 출력합니다. 인벤토리에 아이템이 있다면 인벤토리 정보까지 출력합니다.
- 각 턴이 끝날 때마다 가지고 있는 골드에 비례해서 이자를 얻습니다. 이자는 최대 10골드까지 쌓입니다.

**상점**
- 전투 후에 승리했다면 상점으로 이동(Y, N) -> N을 누르면 전투 시작 화면으로, Y를 누르면 상점 화면으로 갑니다.
- 상점에서는 골드로 아이템을 사거나 가지고 있는 아이템을 팔 수 있습니다.
- 상점에서 떠나면 전투 시작 화면으로 넘어갑니다.

Main :

- 몬스터와 전투 후 상점 방문 여부를 사용자에게 입력받아 선택한 대로 진행되도록 구현
- 재도전과 상점방문 등에서 사용자의 선택을 받아와서 처리하는 부분에 Y 또는 N이 아닐 경우 예외처리

HealthPotion :

### 이재성님 - (Slime)
Monster (팀원이 구현한 클래스)

- Monster 클래스는 모든 몬스터의 공통 속성(이름, 체력, 공격력 등)과 동작을 정의한 부모 클래스입니다.
- 몬스터마다 고유한 드롭 확률과 골드량을 설정할 수 있도록 dropRate와 dropGold 변수를 설계에 포함하였으며,
- 이를 통해 몬스터별로 다른 보상과 드롭 확률을 가질 수 있도록 구현되었습니다.

협업 과정

- 몬스터의 스펙이 캐릭터 레벨에 비례해서 랜덤 생성을 위해, 캐릭터 레벨을 매개변수로 받는 구조를 요청. 
- 랜덤 값 생성을 위한 GetRandomNum(int min, int max) 함수를 Monster 클래스에 구현해달라고 협의.
- dropRate와 dropGold 변수를 활용하여 몬스터마다 고유한 드롭 확률과 보상이 적용되도록 요청.

Slime (대표 예시)

- Slime 몬스터는 Monster 클래스를 상속받아 다음과 같이 초기화됩니다:
- 체력: 기본 값 + 레벨 * 랜덤 값.
- 공격력: 기본 값 + 레벨 * 랜덤 값.
- 드롭률: 10%, 골드 보상량: 4~6 골드.
- 나머지 몬스터도 동일한 구조를 기반으로 되어 있습니다.

Item :

### 박영민님 - (Monster)
몬스터 클래스를 담당해서 구현하면서 헤더 파일과 소스 파일을 분리했다.<br>
왜 헤더 선언을 나눠놓는 게 좋은지 고민한 끝에 헤더 파일에는 정의에 필요한 선언만 포함하고, <br>소스 파일에는 구현에 필요한 헤더 선언을 포함하기로 했다.<br>
그 이유는 **불필요한 헤더 의존성으로 인한 메모리 낭비와 컴파일 시간 증가**를 방지하기 위해서다.

#### 의존성이란? 
여기서 A코드가 작동하려면 무조건 B코드가 있어야 할 때 `A는 B에 의존성을 가진다`고 할 수 있다.<br>
밑의 예시를 보면 Game.h가 Monster.h의 <randmom>이 필요가 없는데도 의존을 하니까 불필요한 의존성이라고 말할 수 있다.
```
// Monster.h
#pragma once
#include <random> // 정의에는 필요 없지만 여기에 포함.

// Game.h
#include "Monster.h" // Game.h도 <random>을 읽음.
```

#### 상속 활용
몬스터 클래스에는 여러 몬스터들이 공통적으로 가지고 있는 속성과 메서드를 넣고 상속하는 식으로 코드를 작성해 중복되는 코드를 줄였다.<br>
예를 들어 getName()이란 메서드는 `return name;`만 해주고 안에 코드 내용이 몬스터 종류에 따라 수정될 일이 없으니 상속으로 처리해주었다.

Item* Monster::DropItem() :
- vector를 이용해 몬스터들이 드랍하는 아이템 목록을 관리했다. (STL: vector)
  
int Monster::GetRandomNum(int min, int max) :
- 몬스터가 드랍하는 아이템의 확률을 구해야 했기 때문에 <random> 함수도 라이브러리에서 가져왔다.

### 최무현님 - (GameManager, MonsterFactory, Shop)

GameManager :
 - 전투와 관련된 로직을 처리하는 클래스, 전투 그 자체 외에도 전투 승리 시 이자를 포함한 골드, 아이템, 경험치 획득, 이번 게임에서 처치한 몬스터의 종류와 수 저장 및 출력 기능이 있다.
 - onordered_map을 이용해 처치한 몬스터의 종류와 수를 저장한다. (STL: onordered_map)
 - 플레이어의 턴이 시작하면 캐릭터의 인벤토리에 있는 아이템을 순회하며 아이템의 IsUsable() 함수를 통해 이 아이템이 현재 사용하기 적합한지 확인 후 아이템을 사용한다. (옵저버)

MonsterFactory :
 - 몬스터를 저장하고 필요한 몬스터를 반환하는 클래스, 원래 GameManager에 있던 몬스터 생성 함수를 따로 클래스로 뺐다. (팩토리 패턴, 단일책임원칙)
 - 처음에는 첫번째 이미지처럼 스위치를 이용해 인스턴스를 생성해 리턴하는 방식이였는데, 이 경우 몬스터가 새로 추가될 때마다 클래스를 수정해야하기 때문에 개방폐쇄원칙을 위반한다.<br>
   실제로 몬스터를 스위치문에 추가하고 랜던값 % 뒤의 숫자를 수정하지 않아 새로 추가한 몬스터가 생성되지 않는 문제가 있었다.<br>
   이 문제를 해결하기 위해 두번째 이미지 처럼 수정했다. AddMonster()를 이용해 main함수에서 몬스터를 추가하면 MonsterFactory 클래스를 수정하지 않고 몬스터가 추가할 수 있다.<br>
   또, 반환할 때는 새로운 인스턴스를 생성하는 것이 아니라 기존에 있던 인스턴스를 초기화해서 반환해서 미미하지만 메모리적으로도 이점이 있다. (개방폐쇄원칙)

### 선우민석님 - (Character, Monster )

Character
 - 플레이어가 조작하게 될 캐릭터의 기능들을 작성한 클래스
 - static Character* 를 변수로 가지며 GetInstance를 통해 접근하며, 객체가 1개이상 생성되는 것을 막는 싱클톤 패턴을 사용하여 플레이어가 조작하는 캐릭터는 하나만 존재하도록 하였다. (싱글톤 패턴)
 - 캐릭터가 가지고 있는 아이템은 vector<Item*> 형태의 변수를 통해 관리하며 Item들의 부모클래스(Item)이 포인터로 접근하여 관리한다.(옵저버 패턴)
 - 캐릭터의 능력치는 Getter와 Setter를 활용하여 값의 확인과 수정이 가능하다.
 - 캐릭터의 상호작용(데미지 받기, 레벨업, 아이템 사용 등)을 메서드로 가지고 있다.

Monster
 - 몬스터들의 틀이 되어줄 부모 클래스
 - 각 몬스터들의 특색이 존재하면서 동시에 몬스터라는 공통점이 존재하기에 여러 몬스터들의 틀이 되어줄 Monster 클래스를 작성하였다.(데코레이트 패턴)
 - 기본적으로 모든 몬스터들이 공통적으로 가지고 있는 능력치와 상호작용의 틀을 작성하고 자식 클래스는 다른 팀원이 작성하였다.

## 상점에서 아이템 구매 및 판매
플레이어는 상점에서 아이템을 구매하거나 판매할 수 있습니다.

![아이템 구매 및 판매 예시](https://github.com/user-attachments/assets/63f337a9-8019-4b30-9110-748fdc215df1)

- 아이템 구매
    - 플레이어는 골드를 사용해서 아이템을 구매 가능
    - 구매 아이템은 인벤토리에 즉시 추가

    - 플레이어는 보유한 아이템을 골드로 판매 가능 **(판매 가격은 구입 원가의 60%!)**
